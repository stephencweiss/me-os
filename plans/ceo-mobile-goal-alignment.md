## Amendment in one paragraph

**Original framing** centered the wedge on **slot-finding** (where to put time) plus widget + audit as supporting loops. **Updated framing:** the **first mobile product** should **help you see whether the week actually matched your goals** — progress, calendar truth vs intent, and a lightweight **weekly alignment** moment — on **iOS-first**. **Slot-finder** (deterministic gaps → suggestions) and a **slot-first widget** stay in the roadmap as **Phase 2**, after the alignment core is trustworthy and dogfooded.

## Step 0 in plain English

**What “Step 0” is:** Before anyone argues about code style or tests, the CEO-style review asks: *Are we solving the right thing, in the right size, with the right ambition?* It’s the “zoom out” step so you don’t beautifully build the wrong product.

**The four “modes” are just attitudes toward scope:**

| Jargon | Plain meaning |
|--------|----------------|
| **Scope expansion** | “Let’s make this bigger and bolder” — you actively add ideas. |
| **Selective expansion** | “Keep the plan, but I’ll **yes/no** a short list of extras.” |
| **Hold scope** | “Don’t add anything — just stress-test what we already agreed.” |
| **Scope reduction** | “Cut to the smallest thing that still wins.” |

**What 0A / 0B / 0C mean (without the letters):**

- **“Are we solving the real problem?”** — Or a stand-in? (e.g. cool slots that never answer *did I actually protect what mattered?*)
- **“What do we already have?”** — So we don’t duplicate MeOS pieces that already work.
- **“Where should this go in a year?”** — So today’s decision doesn’t paint us into a corner.

**What E1–E6 were:** Optional add-ons. **Accept** = in scope for this effort (possibly phased). **Defer** = good idea, not now (track elsewhere).

### Phase 1 — “Alignment mobile” (do this first)

- **iOS app (or mobile-first shell)** whose **main job** is: **this week’s goals ↔ what the calendar shows** (reuse MeOS goal progress, events, summaries — **no second goals system**).
- **E3:** Weekly **alignment** moment: one respectful prompt, dismiss/snooze/backoff — *“Did the week match what you wanted?”* tied to real data.
- **E2 (phase 1):** Freeze **API/contracts** the app needs: auth refresh, week snapshot, goal list + progress, optional **alignment summary** DTO — **before** heavy UI polish.
- **E4:** Per-goal (or per–goal-type) windows so “alignment” isn’t judged against the wrong hours.

### Phase 2 — “Scheduling wedge” (after Phase 1 feels true)

- **Slot-finder** surface (deterministic suggestions, suggest-before-move) + **widget** that can emphasize **next free slot** if you still want that differentiator.
- **E2 (phase 2):** Slot-specific schema / widget extension; same discipline — schema before Swift for *that* feature.
- **E1, E5** as before.

### Deferred (unchanged)

- **E1:** Mandatory “why this slot” — Phase 2.
- **E5:** Widget stale-data UX — Phase 2.

## Step 0A — Premise challenge (amended)

| Question | Take |
|----------|------|
| Right problem? | **Yes**, if framed as **alignment first**: *I set goals — did my calendar reflect them?* Slot-finder is a **tactic**; alignment is the **outcome** you care about on a phone between meetings. |
| Proxy risk? | **Low** if mobile nails alignment; **High** if you ship only slots without a clear “did it work?” story. |
| Do nothing? | You keep opening the **desktop webapp** to answer a **mobile-shaped question** — fine short term, weak validation of “MeOS in your pocket.” |

## Step 0B — Leverage (amended)

- **Already exists:** weekly goals DB, goal ↔ event progress, time-report / summaries, calendar sync, webapp APIs.  
- **Do not rebuild:** duplicate goal models or full calendar editors on mobile.  
- **Do build:** a **thin native (or hybrid) client** + minimal new endpoints if needed for **packaged week views** and **audit state** — then Phase 2 slot-finder API.

## Step 0C-bis — Implementation approaches (amended)

**APPROACH A — Alignment-first mobile (RECOMMENDED now)**  
Ship **read-heavy** iOS MVP: goals, progress, week summary, audit flow; reuse existing APIs; add only what’s missing for **mobile auth** and **snooze/dismiss** persistence.  
Effort: M · Risk: Med · Validates **product thesis** fastest.

**APPROACH B — Contract-first alignment**  
Same as A but **freeze alignment DTOs + OpenAPI/JSON Schema** before Swift UI deep work.  
Effort: M · Risk: Low · Slightly slower start, cleaner parallel work.

**APPROACH C — Slot-first (previous default)**  
Slot-finder + widget before alignment narrative.  
Effort: M/L · Risk: Med · **Deprioritized** — still valid as **Phase 2**.

**RECOMMENDATION:** **A or B** for Phase 1. **Slot-finder (old B/C)** becomes **Phase 2**, not week one.
