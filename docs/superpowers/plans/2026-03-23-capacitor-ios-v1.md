# Capacitor iOS v1 — implementation plan

**Plan date:** 2026-03-23

> **For agentic workers:** Use **subagent-driven-development** or **executing-plans** to implement task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a **reliable MeOS mobile client** on iOS via **Capacitor**, wrapping the existing **Next.js (`web/`)** UI (pnpm workspace package `web`), with a path to **TestFlight** and **App Store** review. **Widgets** (time split + quick categorization) are **explicitly out of scope for v1**; they remain a **phase 2** native follow-on (see RFC PR [#83](https://github.com/stephencweiss/me-os/pull/83)).

**Architecture:** A thin **native iOS shell** (Capacitor) loads the MeOS web app. Prefer a **hosted origin** (`server.url` → production/staging URL) so **App Router + SSR/API routes** keep working without forcing `output: 'export'`. Alternative (documented fallback): static export + simplified hosting if product constraints require fully offline bundles later.

**Tech stack:** `web/` (Next.js 16, App Router), current auth (**NextAuth v5** per `web/package.json`). If MeOS migrates to **Clerk** (or adds **Sign in with Apple** alongside Google), the same Capacitor constraints apply: **cookie / session domain**, **OAuth redirect URIs**, and **universal links** must be updated in one place.

**Spec:** None required for v1 shell; optional future spec: `docs/superpowers/specs/…-ios-capacitor-design.md` if UI/flow splits web vs mobile.

**Related:** `plans/ios-migration-*.md` on branch `sw-ios-migration-plans` (RFC comparing stacks). **Decision:** v1 = **Capacitor**; native Swift + WidgetKit = **later**.

**Tracking:** https://github.com/stephencweiss/me-os/issues/98

## Constraints (do not skip)

| Topic | Requirement |
|--------|-------------|
| **Background refresh** | iOS does not guarantee continuous sync. Plan for **refresh on resume**, optional **push** (user-visible or silent within Apple limits), and **Background App Refresh** as best-effort only. |
| **Rich offline** | **Out of scope for v1** per product direction; document if that changes. |
| **App Store / auth** | If the app offers **third-party login** (e.g. Google) for account creation/access, plan **Sign in with Apple** on iOS unless a documented **Apple exemption** applies. Implement in the chosen auth provider (Clerk/NextAuth) before submission. |
| **Privacy & accounts** | Privacy nutrition labels, privacy policy URL, and **account deletion** flow must match actual data practices. |

### Reminder: URLs & Apple (when you pick this up)

1. **Staging / prod URLs** — Choose HTTPS origins; set `server.url` in Capacitor for builds that should load the deployed site. For day-to-day dev, you can point at `http://localhost:3000` only where iOS ATS / dev settings allow (or use tunnel tools).
2. **[Apple Developer](https://developer.apple.com)** — Membership → **Certificates, Identifiers & Profiles** → **Identifiers** → register an **App ID** (bundle ID, e.g. `com.yourname.meos`). Enable capabilities you need (Push, Sign in with Apple, etc.) when you add them.
3. **[App Store Connect](https://appstoreconnect.apple.com)** — **My Apps** → **+** → **New App**; match the bundle ID; fill minimal metadata (you can refine before submission).
4. **Xcode signing** — Open `web/ios/App.xcworkspace` (after Capacitor generates it), select the app target → **Signing & Capabilities** → Team + **Automatically manage signing** (uses the App ID from step 2).

## Rollback / risk notes

- **Hosted `server.url`:** If production is down, shell shows error — consider offline **friendly error UI** in web app.
- **Auth provider migration (e.g. NextAuth → Clerk):** Re-validate **all** OAuth redirects and cookie names before shipping a store build.
- **Committing `ios/`:** Team must agree on **generated project in git** vs **regenerate in CI**; pick one pattern to avoid drift.
